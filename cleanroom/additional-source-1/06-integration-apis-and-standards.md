# Additional Source 1 — integration, APIs, and standards (advertised)

## Open API posture

Netcracker publicly emphasizes **open APIs** and **standard** interface frameworks so CSPs can integrate **partners**, **BSS**, and **orchestrators**. Marketing references **industry API frameworks** (examples cited in public materials include **MEF LSO**-style API families for service provider interoperability).

## BSS coupling

The portfolio is explicitly **BSS-adjacent**: inventory and service data are intended to feed **commercial** processes (quoting, ordering, billing) in a **converged** operator stack—far broader than Source A’s typical **engineering automation** focus.

## DevOps and CI/CD

Vendor messaging ties **microservice** delivery to **Agile/DevOps** practices—relevant for how upgrades and customizations are **sold**, not something validated in this clean-room folder.

## Integration adapters

**Service Activation+** and orchestration collateral mention **adapters** and **templates** to accelerate **partner** services and **multivendor** equipment—integration depth is a **commercial differentiator** versus downloadable open tools.

## What we do not document here

Specific **REST** path catalogs, **authentication** schemes, or **SDK** details belong in **vendor** documentation under license—not reproduced in this exercise.
